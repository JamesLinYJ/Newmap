面向本科生的 Agent 开发讲稿

主题：Agent 是什么、为什么重要、以及我们应该如何从 0 到 1 开发一个可用的 Agent

授课对象：本科生

建议时长：45 到 60 分钟

课程目标：让学生理解 Agent 的基本概念、核心组成、开发流程、常见误区，以及一个最小可用 Agent 的实现思路。

一、开场：为什么今天大家都在讲 Agent

同学们，最近大家经常会听到一个词，叫 Agent。很多产品都说自己不是普通聊天机器人，而是 Agent。那它们到底有什么不同？我们今天不只是讲概念，更重要的是搞清楚：如果让我们自己开发一个 Agent，应该从哪里开始，怎么设计，怎么验证它是真的有用，而不是“看起来很聪明，实际上不可靠”。

我希望大家今天带走三件事：

  1. 知道 Agent 和普通问答模型的区别。
  2. 知道一个 Agent 至少由哪些模块构成。
  3. 知道怎样开发一个最小可用版本，并持续把它变得更可靠。

二、什么是 Agent

一个比较容易理解的定义是：Agent 是一个能够围绕目标进行感知、推理、决策和行动的软件系统。

普通大模型更像“回答问题的人”，你问一句，它答一句。而 Agent 更像“接任务的人”，它不仅回答，还会自己拆步骤、调用工具、读取信息、执行操作，并根据执行结果继续调整下一步。

1. 普通 LLM 和 Agent 的区别

  • 普通 LLM：更偏“生成答案”。
  • Agent：更偏“完成任务”。
  • 普通 LLM：输入通常是一个 prompt。
  • Agent：输入通常是目标、上下文、工具、状态、记忆和反馈。
  • 普通 LLM：输出是一段文本。
  • Agent：输出可能是文本、计划、动作、文件、数据库写入、API 调用结果等。

2. Agent 的最小闭环

我们可以把 Agent 简化成一个循环：

用户给目标
↓
Agent 理解任务
↓
Agent 选择动作
↓
调用工具执行
↓
读取执行结果
↓
判断是否完成
↓
完成就输出，否则继续循环

所以，Agent 的关键不是“会不会说”，而是“能不能围绕目标稳定完成闭环”。

三、一个 Agent 由哪些核心模块组成

1. 模型层

模型层就是大语言模型本身，比如 GPT、Claude、Gemini、Qwen 等。它负责理解语言、生成计划、决定下一步做什么。

课堂强调点：模型很重要，但模型不是 Agent 的全部。真正让 Agent 有执行力的，是后面的工具层和状态层。

2. Prompt 层

Prompt 可以理解成给模型的“工作说明书”。它通常包括：

  • 系统角色：你是谁。
  • 任务目标：你要完成什么。
  • 规则约束：哪些事情能做，哪些事情不能做。
  • 输出格式：你要按什么结构返回。

很多同学一开始做 Agent，会过度迷信 prompt engineering。其实 prompt 很重要，但如果没有工具、状态和验证，光靠 prompt 往往不够稳定。

3. 工具层

工具层决定 Agent 能不能“动手”。常见工具包括：

  • 搜索工具：查资料、查网页、查数据库。
  • 代码工具：运行代码、执行脚本、读写文件。
  • 业务工具：下单、查询库存、发送邮件、审批流程。
  • 知识工具：RAG 检索、企业知识库、FAQ 系统。

没有工具，Agent 只能空想。有了工具，Agent 才能真正接近“完成任务”。

4. 状态层

状态层用于记录 Agent 当前处于哪个阶段。例如：

  • 任务已经解析到哪一步。
  • 哪些工具已经调用过。
  • 是否已经拿到中间结果。
  • 有没有报错。
  • 最终是否完成。

如果没有状态层，Agent 就像一个没有短期记忆的人，每次都像重新开始。

5. 记忆层

记忆层一般分为两类：

  • 短期记忆：当前会话里的上下文。
  • 长期记忆：跨会话保存的偏好、历史任务、用户信息。

对于初学者来说，可以先不用做得太复杂。先把“当前任务状态”做好，再逐步扩展到长期记忆。

6. 观察与评估层

一个真正可用的 Agent，一定要能观察和评估。至少要记录：

  • 用户输入了什么。
  • 模型做了什么判断。
  • 调用了哪些工具。
  • 中间结果是什么。
  • 在哪一步失败了。
  • 最终结果是否让用户满意。

如果没有这些日志和指标，Agent 出错时你会完全不知道问题出在哪里。

四、Agent 开发的典型流程

第一步：先定义任务，不要先上模型

开发 Agent 最常见的错误，是一上来就选模型、写 prompt、接 SDK。其实第一步应该是：先把任务定义清楚。

我们要回答这些问题：

  • 这个 Agent 是帮谁做事？
  • 它要完成的目标是什么？
  • 成功的标准是什么？
  • 失败的边界是什么？
  • 它需要调用哪些工具？

比如“做一个校园教务 Agent”，这句话太大了。更合适的定义可能是：帮助学生查询课程信息、生成选课建议、提醒时间冲突，但不能替学生提交最终选课操作。

第二步：把任务拆成状态机或工作流

把任务拆成几个可控节点，比让模型“自由发挥”更可靠。一个常见结构是：

接收任务
→ 解析意图
→ 选择工具
→ 执行工具
→ 验证结果
→ 输出答案或继续下一步

如果任务更复杂，还可以拆成多 Agent 协作，但对本科生项目来说，通常先从单 Agent 工作流做起就够了。

第三步：设计工具接口

工具不是越多越好，而是越清晰越好。一个好的工具接口要满足三点：

  1. 输入参数明确。
  2. 输出结构稳定。
  3. 错误信息可诊断。

比如，不要让工具只返回“失败了”。要返回更具体的信息，例如：权限不足、超时、资源不存在、参数错误。

第四步：让模型只做它擅长的部分

模型擅长的是理解语义、抽取意图、生成计划、解释结果。模型不擅长的是精确计算、稳定检索、严格执行和长期一致性。

所以一个好的原则是：

让模型负责“判断和组织”，让工具负责“计算和执行”。

第五步：加验证，不要直接相信模型

Agent 开发里很重要的一件事，就是不要把模型输出当真理。需要加验证机制，例如：

  • 输出 JSON 时校验 schema。
  • 调用工具前校验参数是否合法。
  • 工具返回后检查结果是否为空。
  • 关键任务加入人工确认。

课堂强调点：Agent 的核心能力不是“会说服你它对了”，而是“它真的可验证、可控制、可恢复”。

五、一个最小可用 Agent 的开发示例

下面我们用一个简单例子帮助大家理解。假设我们要做一个“校园通知整理 Agent”。

需求

  • 输入：学校官网通知链接或通知文本。
  • 输出：摘要、分类、截止日期、是否与学生本人相关。
  • 扩展：可以把任务加入待办清单。

最小架构

前端页面
↓
后端 API
↓
Agent Runtime
├─ Prompt
├─ LLM
├─ 工具：网页抓取 / 分类 / 日历写入
└─ 状态存储：任务状态、日志、结果

最小伪代码

user_input = "帮我整理这条通知是否重要"

intent = llm.parse_intent(user_input)

if intent.requires_webpage:
    notice = tools.fetch_page(intent.url)
else:
    notice = intent.text

summary = llm.summarize(notice)
category = llm.classify(notice)
deadline = llm.extract_deadline(notice)

validated = validate(summary, category, deadline)

if validated:
    return result
else:
    return error

大家会发现，这里并不复杂。一个最小可用 Agent 的关键不是一开始就做得特别炫，而是先让这条链路真实可运行。

六、Agent 开发中的常见误区

误区一：把 Agent 当成“更强的聊天机器人”

如果只是聊天体验好一点，那不一定需要 Agent。只有当任务需要规划、调用工具、执行动作时，Agent 才真正有价值。

误区二：工具接得越多越强

工具太多，模型的决策空间会变大，反而更容易选错。初期要控制工具数量，让每个工具边界清晰。

误区三：只做 Demo，不做观测

很多 Demo 看上去成功率很高，但一上真实用户就出问题。原因通常不是模型不够强，而是没有日志、没有状态记录、没有失败恢复。

误区四：把多 Agent 当作高级感

多 Agent 不等于更高级。很多问题单 Agent 就能解决，多 Agent 反而会增加复杂性、延迟和调试难度。

七、如何评估一个 Agent 做得好不好

评估不能只看“回答像不像人”，还要看是否稳定完成任务。可以从以下几个维度评估：

  1. 任务成功率：任务是否真的完成。
  2. 工具调用正确率：是否选对工具、参数是否正确。
  3. 延迟：完成任务需要多久。
  4. 成本：每次运行用了多少 tokens、多少 API 成本。
  5. 鲁棒性：输入不规范时是否还能处理。
  6. 可恢复性：中途失败后能否重试或回滚。

八、给本科生的开发建议

  1. 先做单 Agent，不要一开始做多 Agent。
  2. 先做 1 到 2 个高价值工具，不要铺太大。
  3. 先做可观测、可验证，再追求“像真人一样自然”。
  4. 先把任务完成率做好，再考虑复杂记忆和自动规划。
  5. 项目汇报时，多展示“任务闭环”和“失败处理”，这比只展示聊天界面更有说服力。

九、课堂总结

今天我们讲的核心可以概括成一句话：

Agent 不是一个更会聊天的模型，而是一个围绕目标进行决策、执行和反馈闭环的系统。

开发 Agent 时，我们真正要关注的是：

  • 目标是否清晰。
  • 工作流是否可控。
  • 工具是否可靠。
  • 状态是否可追踪。
  • 结果是否可验证。

如果大家以后要做课程项目，我建议从一个非常具体的小场景入手，比如校园通知、课程答疑、实验记录整理、社团报名助手。把一个小 Agent 做扎实，比做一个华而不实的大 Agent 更重要。

十、课后思考题

  1. 如果让你做一个“实验室值班 Agent”，它需要哪些工具？
  2. 如果一个 Agent 可以自动帮用户发邮件，为什么必须增加确认机制？
  3. 在你看来，哪些任务适合单 Agent，哪些任务可能需要多 Agent？
  4. 如果 Agent 输出错了，你会如何定位问题出在 prompt、模型、工具还是状态管理？

附：适合课堂直接讲的收尾话术

最后我想强调，Agent 开发并不是“把一个大模型 API 接上去”这么简单。它本质上是软件工程、人工智能和业务流程设计的结合。谁能把模型、工具、状态、验证和用户体验一起做好，谁才能做出真正可用的 Agent。

希望大家听完这节课之后，不只是会说“我知道 Agent 是什么”，而是真的知道：如果给我一个实际问题，我应该怎样把它一步一步做成一个 Agent 系统。
