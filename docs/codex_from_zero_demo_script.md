# 用 Codex 从 0 构建 AI GIS 地图分析工作台详细演示稿

## 一、演示总目标

各位老师、各位同事好，今天这场演示我想展示的不是一个单点功能，也不是简单证明“AI 可以写代码”。我真正想展示的是：如果我们从一个完全空的项目目录开始，如何把 Codex 当成一个工程搭档，完成一个 AI GIS 地图分析工作台从需求沟通、技术选型、架构设计、任务拆解、代码实现、联调验证到最终展示的完整过程。

这个演示的核心场景是“自然语言调用地图分析”。传统 GIS 系统通常要求用户先理解图层、工具、空间关系和参数设置，例如先选择医院图层、再选择机场坐标、再设置 3 公里缓冲区、再做空间筛选。对非 GIS 专业用户来说，这个过程门槛比较高。我们希望做的工作台，是让用户直接输入一句中文，例如：

```text
查询东京羽田机场 3 公里范围内的医院
```

系统自动理解这句话里的地点、距离、目标对象和输出形式，然后决定要调用哪些工具：先解析“东京羽田机场”的位置，再查询 3 公里范围内的医院 POI，再生成 GeoJSON，最后在地图上展示，并用中文解释结果。

因此，这个演示不是单纯展示一个聊天机器人，而是展示一个“AI + GIS 工具链”的工程闭环。这里面有自然语言理解，有工具调用，有地图渲染，有数据格式，有前后端联动，也有项目工程化启动方式。

我会把演示分成九个环节：

```text
1. 先和 Codex 沟通需求
2. 让 Codex 推荐技术框架
3. 让 Codex 设计系统架构
4. 让 Codex 拆开发任务
5. 从空目录创建项目
6. 实现后端 Agent 工具接口
7. 实现前端地图工作台
8. 进行联调和演示
9. 说明风险预案和后续扩展
```

如果只看最终效果，大家可能会觉得这是一个普通 Demo。但我更希望大家关注过程：Codex 如何把一个模糊需求变成可执行任务，如何在技术选型时给出比较，如何在实现中补齐前后端，如何在联调时根据错误继续修复。这才是 Codex 对工程效率真正有价值的地方。

## 二、演示前准备

正式演示之前，我会准备两个目录或两个分支。

第一个是空目录，用于现场从 0 开始，让大家看到 Codex 如何搭项目骨架、生成代码和解释方案。这个目录可以叫：

```text
ai-gis-demo-empty
```

第二个是完成版目录或完成分支，用于兜底展示完整效果。这个目录可以叫：

```text
ai-gis-demo-finished
```

为什么要准备两个版本？因为现场演示会受很多因素影响，例如 npm 安装慢、Python 包安装慢、地图瓦片加载慢、模型响应慢、外部接口限流。如果所有内容都押在实时生成上，风险太高。专业的演示应该有实时路径，也有完成路径。

现场我可以这样说：

```text
今天我会先从空目录演示 Codex 如何搭建项目，让大家看到从 0 到 1 的过程。为了避免现场网络或依赖安装影响演示节奏，我也准备了一个完成版本。如果某个依赖加载较慢，我们会直接切到完成版本，把核心功能完整展示出来。
```

这样说不是降低演示可信度，而是体现工程意识：我们既展示创造过程，也保证最终效果可见。

### 预期 Codex 输出

Codex 理想情况下会输出几类内容。

第一类是用户故事。例如：

```text
作为业务分析人员，我希望直接输入中文空间问题，不需要手动选择 GIS 工具，也能得到地图结果和文字解释。
```

第二类是核心功能。例如：

```text
1. 中文自然语言输入
2. 地点解析
3. POI 查询
4. GeoJSON 结果生成
5. 地图展示
```

```text

3. POI 数据可能不完整

```

### 讲解重点

我会解释：这一步看似没有写代码，但非常关键。因为它让 Codex 先理解项目边界。GIS 项目尤其不能靠猜，因为它涉及数据、坐标、图层、空间关系、外部服务和结果呈现。如果一开始没有明确“明天演示只做 MVP”，Codex 可能会把权限系统、数据库管理、异步任务、QGIS 服务发布都放进第一版，反而影响演示。

这里我会强调一句话：

```text
先让 Codex 理解问题，再让它解决问题。
```

这句话是第一部分的核心。

## 四、第二阶段：让 Codex 推荐技术框架

### 现场话术

需求拆完后，我会说：

```text
现在需求边界有了，但还不能直接写代码。下一步要做技术选型。这里我不直接告诉 Codex 用什么，而是让它比较方案。这样可以看到它不仅能写代码，也能参与架构判断。
```

输入 Prompt：

```text
基于刚才的需求，请你推荐适合明天演示的技术架构。

要求：
- 快速开发
- 本地能跑
- 前端界面要好看
- 后端要能扩展成真实 Agent 工具调用
- 地图要能展示 GeoJSON
- 先允许 mock 数据，后续可接真实 OSM / Nominatim / Overpass

请比较三种方案：
1. Streamlit 快速 Demo
2. React + FastAPI
3. Next.js 全栈

从开发速度、演示效果、可扩展性、地图能力、后续生产化角度比较，并给出推荐。
```

### 技术方案对比讲稿

Codex 可能会比较三种方案。我会顺着它的输出解释。

Streamlit 的优点是快。如果只是给内部看一个简单原型，Streamlit 非常合适。它可以很快写出输入框、按钮、地图和表格。但 Streamlit 的问题是，它更像数据应用，不太适合复杂前端交互。如果我们要做类似工作台的页面，要做聊天区、地图区、结果区、图层管理、工具调用过程和状态流，Streamlit 会比较吃力。

Next.js 的优点是全栈能力强，页面工程能力好，后续产品化也不错。但这个项目的 GIS 处理和 Agent 工具更多在 Python 生态里，例如 FastAPI、GeoJSON、Shapely、PostGIS、QGIS、LangChain、Deep Agents。这些能力放在 Python 后端更自然。如果全部放进 Next.js，会让后端 GIS 能力不够顺手。

React + FastAPI 是比较平衡的方案。React 专注前端体验，FastAPI 专注后端 API 和工具编排。前端可以做成正式产品感，后端可以自然接 Python GIS 生态。两边职责清楚，也方便演示和后续扩展。

所以我会推荐：

```text
前端：React + Vite + TypeScript
地图：MapLibre GL
UI：Tailwind CSS 或项目内 CSS
动画：Framer Motion
后端：FastAPI
Agent：Deep Agents / LangChain Tools
数据格式：GeoJSON
演示数据：mock provider
真实扩展：Nominatim / Overpass / PostGIS / QGIS
```

### 为什么不是直接上完整生产架构

这里我会补一句：明天演示不应该一开始就上完整生产架构。比如 PostgreSQL、PostGIS、QGIS Server、异步队列、权限系统都很重要，但不是第一版必须全部展示。演示阶段要先把核心闭环跑通：中文输入、工具调用、地图结果、中文解释。等闭环稳定后，再接真实服务。

这也是工程里常见的分层思路：先验证核心链路，再扩大基础设施。

## 五、第三阶段：让 Codex 设计系统架构

### 现场 Prompt

技术栈确定后，我会让 Codex 设计架构：

```text
请基于 React + FastAPI 设计系统架构。
要求包括：
- 前端页面结构
- 后端 API
- Agent 工具列表
- 数据流
- 状态流
- 项目目录结构
- MVP 开发顺序
```

### 架构讲解稿

我希望 Codex 输出四层架构。

第一层是前端交互层。包括：

```text
ChatPanel：中文输入、对话结果、工具过程
MapPanel：MapLibre 地图、GeoJSON 图层、高亮结果
ResultPanel：结果摘要、下载、下一步建议
LayerPanel：图层列表、图层开关、上传入口
```

第二层是 API 与任务层。包括：

```text
POST /api/chat：提交中文问题
GET /api/runs/{id}：查看运行状态
GET /api/runs/{id}/events：查看事件流
GET /api/artifacts/{id}.geojson：下载结果
```

第三层是 Agent 工具层。包括：

```text
geocode_place：解析地点
search_poi：查询兴趣点
make_buffer：生成缓冲区
intersect_geojson：空间相交
export_geojson：导出结果
summarize_result：生成中文总结
```

第四层是数据与服务层。演示阶段是 mock provider，后续是真实 provider：

```text
MockProvider：稳定演示
Nominatim：真实地理编码
Overpass：真实 OSM POI 查询
PostGIS：空间数据存储和分析
QGIS：复杂空间处理和发布
```

### 数据流讲解

我会用一句问题解释数据流：

```text
查询东京羽田机场 3 公里范围内的医院
```

它的执行链路是：

```text
用户输入
→ FastAPI 接收任务
→ Agent 判断任务类型
→ geocode_place 解析羽田机场坐标
→ search_poi 查询 3 公里范围内医院
→ export_geojson 生成地图结果
→ summarize_result 生成中文说明
→ React 前端展示对话、工具过程和地图图层
```

这条链路说明：模型不是直接编答案，而是通过工具完成任务。这样系统才可解释、可验证、可扩展。

## 八、第六阶段：实现后端 Agent 工具接口

### 现场 Prompt

```text
继续实现 FastAPI 后端。
要求：
- POST /api/chat 接收 JSON：{ "message": "查询东京羽田机场3公里范围内的医院" }
- 返回 answer、toolCalls、geojson 三部分
- 工具链包括 geocode_place、search_poi、export_geojson
- geocode_place 和 search_poi 先使用 mock provider
- 代码中要把 provider 抽象出来，后续方便替换成 Nominatim 和 Overpass
```

### 讲解重点

我会强调后端返回结构要稳定。比如：

```json
{
  "answer": "已找到东京羽田机场 3 公里范围内的医院点位。",
  "toolCalls": [
    {"tool": "geocode_place", "status": "completed", "summary": "解析东京羽田机场坐标"},
    {"tool": "search_poi", "status": "completed", "summary": "查询 3 公里范围内医院"},
    {"tool": "export_geojson", "status": "completed", "summary": "生成地图结果"}
  ],
  "geojson": {
    "type": "FeatureCollection",
    "features": []
  }
}
```

这里我会说：

```text
这个接口不是随便返回一段文本，而是返回结构化结果。前端可以根据 answer 显示中文解释，根据 toolCalls 显示执行过程，根据 geojson 渲染地图。这样前后端就不会互相猜。
```

### 为什么先 mock

我会解释：

```text
mock provider 的作用不是造假，而是把演示链路稳定下来。我们先验证工具调用、地图展示和交互体验；后续只要把 provider 换成真实的 Nominatim 和 Overpass，前端和 Agent 结构不用推倒重来。
```

## 十三、可能被问到的问题与回答

### 问题 1：这是不是只是一个 mock Demo？

回答：

```text
演示阶段确实使用 mock provider，但 mock 的目的是保证现场稳定。系统结构已经把 provider 抽象出来，后续可以替换成 Nominatim、Overpass、PostGIS 或 QGIS。关键不是 mock 数据本身，而是自然语言、工具调用、GeoJSON 输出和地图展示这条链路已经跑通。
```

### 问题 2：为什么不用现成 GIS 软件直接做？

回答：

```text
现成 GIS 软件适合专业人员操作，但它要求用户知道图层、工具和参数。这个工作台面向的是自然语言入口，让非专业用户也能提出空间问题。它不是替代 GIS 软件，而是把 GIS 能力封装成更容易使用的智能入口。
```

### 问题 3：AI 会不会乱调用工具？

回答：

```text
不会让模型完全自由执行。工程上会给它工具清单、参数 schema、数据源目录和校验规则。模型可以推理选择工具，但最终只能调用系统注册过的工具，引用真实存在的数据源。这样既有智能性，也有边界。
```

### 问题 4：后续怎么接真实数据？

回答：

```text
后续可以分三步。第一，把 mock geosearch 替换成 Nominatim。第二，把 mock POI 替换成 Overpass。第三，把长期管理数据接入 PostGIS，并在需要时接 QGIS 做复杂空间处理。前端和 Agent 工具接口可以保持基本不变。
```

### 问题 5：Codex 生成的代码可靠吗？

回答：

```text
Codex 生成代码以后仍然需要工程验证。我的演示流程里专门包括测试和联调步骤，让 Codex 运行命令、读取错误、修复问题。我们不是盲目信任生成结果，而是把 Codex 纳入正常工程流程。
```

## 十四、最终总结

最后我会这样收尾：

```text
今天这场演示想说明的是，Codex 不只是代码补全工具，也不只是聊天助手。它可以参与一个项目从 0 到 1 的完整过程：先帮我们澄清需求，再比较技术方案，再设计系统架构，再拆开发任务，再生成代码，再参与联调和修复。

对 AI GIS 工作台这种项目来说，真正的难点不是某一个页面，也不是某一个接口，而是把自然语言、地理数据、空间分析、地图展示和工程部署串成一条稳定链路。Codex 的价值就在于，它能帮助我们把这条链路拆清楚、建起来、跑通，并且让后续扩展有明确方向。

所以，如果明天从 0 开始做这样一个项目，我不会把 Codex 当成单纯代码生成器，而会把它当成工程搭档。它帮助我们把模糊想法变成需求，把需求变成架构，把架构变成代码，把代码变成可以演示的系统。
```

## 二十、如何体现“子智能体分工”

如果你想在演示里讲子智能体，可以这样说：

```text
这个项目后续可以拆成多个子智能体协作。Supervisor 负责理解用户问题和分配任务。Spatial Analyst 负责地点解析、POI 查询、缓冲区和相交分析。QGIS Operator 负责复杂 GIS 处理和模型执行。Publisher 负责把结果导出成 GeoJSON 或发布成地图服务。前端只看到一个统一助手，但内部可以按职责分工。
```

可以展示一个文字结构：

```text
Supervisor
```
