# 用 Codex 从 0 构建 AI GIS 地图分析工作台纯讲述稿

## 开场

今天我想讲的不是“AI 能不能帮我写代码”，而是一个更接近真实工程的问题：如果我要从零开始，做一个 AI GIS 地图分析工作台，我应该怎样和 Codex 协作，怎样让它不只是写几段代码，而是参与计划、评审、分工、实现、联调和交付。

这个项目的目标可以先说得很简单。用户用中文提出空间分析问题，比如“查询东京羽田机场 3 公里范围内的医院”，系统能够理解地点、距离和查询对象，自动调用地图、地理检索和空间分析工具，最后把结果展示在地图上，并给出一段中文解释。

但真正开始做的时候就会发现，这不是一个普通聊天框，也不是一个简单地图页面。它里面有前端、有后端、有地图服务、有空间数据、有模型调用、有任务编排、有运行状态，还有一堆看起来不大、但实际很容易互相打架的细节。地点从哪里解析，医院数据从哪里来，地图怎么展示，结果要不要保存，用户能不能上传图层，模型出错怎么办，任务过程要不要展示，后续能不能发布地图服务，这些问题如果没有先想清楚，直接写代码很容易散。

所以我不会一开始就让 Codex 写页面，也不会一开始就让它接接口。越是复杂系统，越不能把 Codex 当成纯粹的代码工人。最稳的做法，是先让它生成计划，再让它 review 计划，再由主智能体指挥子智能体分工实现，最后统一 review、测试、联调和交付。

这套流程听起来比“直接写代码”慢一点，但实际是在省时间。因为真正昂贵的，从来不是第一版代码，而是第一版代码跑起来以后才发现，图层是写死的，fallback 是乱跳的，状态是说不清的，前端和后端各自以为自己是对的。我们提前把这些问题拎出来，后面 Codex 才能像一个靠谱的工程协作者，而不是一个只会快速产出碎片代码的执行器。

## 先让 Codex 生成计划

第一步，我会先把最原始的需求交给 Codex：我想做一个 AI GIS 地图分析工作台，用户可以用自然语言调用地图和空间分析能力，希望明天能有一个可演示版本，同时后续还能扩展成真实系统。

这时候我不会让它马上写代码，而是让它先生成一份可以执行的项目计划。这个计划不是为了好看，也不是为了做流程表，而是让 Codex 把它对项目的理解先摆出来。它准备先做什么、后做什么，哪些依赖要先解决，哪些边界要先确认，哪些地方必须留接口，哪些地方不能为了演示先写死，这些都应该先讲清楚。

我会要求这份计划至少覆盖几个关键维度。第一是产品范围。它要说清楚这次做的不是一个泛泛的地图页面，而是一个 AI GIS 工作台，重点是“分析”和“协同”，不是纯展示。第二是最小可演示链路。明天要演示，不可能把所有 GIS 能力都做完，所以要先确定一条稳定主链路，比如用户输入一个中文空间查询，系统解析地点，查询周边 POI，生成 GeoJSON，地图上展示结果，右侧给出摘要。第三是技术路线。前端、后端、Agent、地图、数据和部署分别怎么选，为什么这样选。第四是风险边界。哪些地方允许降级，哪些地方绝对不能偷偷 fallback；哪些内容可以本地缓存，哪些内容必须以服务端状态为准；哪些数据可以为空，哪些数据缺失时必须明确告诉用户。

这个计划的价值，不在于它写得像不像文档，而在于它暴露了 Codex 的思考。只要它先把路线讲出来，我就能很快看出它有没有真正理解任务。比如它是不是把地图交互和数据分析的优先级搞反了，是不是把一次性 Demo 方案误当成可扩展架构，是不是只想着页面好看，却没有考虑结果怎么保存、任务怎么恢复。

计划里还要明确验收标准。比如不是“页面上能显示地图”就算完成，而是要能从一次自然语言请求开始，创建 run，产生 event，调用工具，生成 artifact，前端加载 GeoJSON 并显示在地图上。也不是“模型能回答”就算完成，而是要保证回答背后有真实工具调用、有可追溯结果、有失败说明。

这一步看似只是规划，实际上是在把“我想做一个 GIS Agent”这句模糊需求，翻译成工程能执行的语言。只有先把地基画出来，后面的生成才不会变成一堆看似完整、其实彼此冲突的实现。

## 再让 Codex review 计划

计划先把路画出来，但画出来不代表能走。所以第二步，我会让 Codex review 自己的计划。

这个 review 不是走形式，也不是让它礼貌地点头，而是让它专门去找计划里以后一定会变成 bug 的地方。我会让它站在更严格的工程角度检查：这个计划有没有过度设计，哪些地方明天演示来不及，哪些地方可能是假 Demo，哪些地方容易写死，哪些地方将来扩展会痛苦。

第一类重点风险，是图层写死。GIS 项目里很容易为了演示快，直接把几个图层 ID、样式配置、数据源地址写进组件里。短期看一切正常，但一旦数据源切换、城市切换、图层权限切换，整个页面就变成只能跑在这一版数据上的临时方案。所以 review 的时候要明确：凡是图层顺序、图层可见性、图层样式、图层数据源，只要已经属于业务配置，就不能继续停留在组件常量里，而应该变成后台 catalog 或运行时配置。

第二类风险，是 fallback 边界不清。很多系统遇到异常就本能地“先兜底再说”，但在工作台这种场景里，fallback 不是越多越好，而是越清楚越好。网络失败时显示“暂时无法连接服务”可以接受，但不能把缓存结果悄悄冒充最新分析结果。地图资源无法加载时显示占位层可以接受，但不能自动切到另一套含义不一致的默认图层。模型没有给出可执行计划时，可以进入修复循环，但不能偷偷换一套规则执行，然后把补救过程包装成正常成功。

第三类风险，是状态不清。一个“开始分析”按钮背后其实可能有很多状态：用户是否已选区域，当前图层是否已加载，模型任务是否已提交，结果是否在流式返回，上一次错误是否还要保留，当前输入是不是用户正在编辑的草稿。review 计划时，我会让 Codex 检查这些状态有没有混在一起。如果计划里只写“显示加载中”和“显示结果”，那就太粗了；真正可靠的计划，应该区分服务端事实态、前端编辑态、地图临时态和 debug 诊断态。

第四类风险，是最小可演示范围被做大。AI GIS 工作台可以做的东西很多，图层管理、地理编码、POI 查询、缓冲区、相交分析、上传数据、发布服务、历史回放、审批流程都很重要，但明天演示不可能全部做满。review 要帮我们收敛：先打通核心闭环，再扩展能力。否则很容易每个功能都有一点，但没有一条链路真正稳定。

所以 Codex review 计划的目标，不是检查有没有把功能写全，而是检查这个计划是不是足够“能维护”。它要能回答：谁是状态的单一事实来源，哪一层可以降级，哪一层不能降级，哪些内容是配置驱动，哪些内容绝不能写死，一旦失败，系统该怎么恢复，而不是只会报错。

经过这一轮 review，计划就会从“能做什么”变成“怎样做才不乱”。边界确认之后，下一步才是真正把任务拆开。

## 指挥子智能体分工干活

进入实现阶段以后，我不会让一个 Codex 从头写到尾。更接近真实工程的方式，是让主智能体像技术负责人一样，先拆任务，再指挥多个子智能体并行干活。

主智能体的作用不是替所有子智能体做事，而是把复杂工程变成可管理的协作秩序。它负责拆、派、并、收、改、审。先拆任务，确认哪些模块可以并行；再派给子智能体，明确每个子智能体的职责、输入、输出和不能改的范围；并行推进时持续看接口边界；结果回来后统一收敛；发现冲突再重新拆小任务；最后做集成 review。

比如这个 AI GIS 项目里，主智能体会先把任务拆成几个工作面。后端子智能体负责 API、运行状态、数据库和事件流；前端子智能体负责聊天面板、地图交互、图层面板和结果展示；GIS 子智能体负责地理检索、POI 查询、GeoJSON 产物、坐标和空间分析边界；测试子智能体负责主链路、边界场景和回归覆盖；部署子智能体负责本地启动、Docker、环境变量和服务器部署；文档子智能体负责把架构、接口、运行方式和演示稿整理清楚。

但这里有一个前提，不能让子智能体各写各的。主智能体必须先定义清楚边界。比如后端 API 返回什么字段，前端只能消费什么状态，工具层必须用什么参数结构，artifact 怎么命名，错误怎么表达，这些都要提前约定。否则多个子智能体同时干活，很容易写出几个局部都对、合起来不通的模块。

具体来说，主智能体会先给后端子智能体一个边界：先把数据模型、API 契约、错误码和异步任务状态机定下来，先不要碰前端展示。给前端子智能体的边界则是：根据后端契约先搭页面骨架、状态容器和交互占位，不能自己发明业务规则，也不能把工具过程写死成假卡片。给 GIS 子智能体的边界是：只回答空间能力怎么封装，坐标转换由谁做，POI 查询返回什么结构，GeoJSON 怎么表达，哪些失败必须明确报错。给测试子智能体的边界是：围绕真实主链路写场景，不能为了测试通过改业务逻辑。给部署子智能体的边界是：只处理环境、构建、脚本和健康检查，不混进业务实现。

这样的分工不是平均分活，而是把“互相依赖最少、接口最明确”的部分并行出去。后端和前端可以同时开始，但中间靠接口文档和共享类型对齐，不会等到最后才发现字段名不一样、状态机不一致、分页语义也不一致。

GIS 子智能体通常要单独抽出来，因为它最容易成为隐性复杂度中心。主智能体不会让它顺手把地图页面也做了，而是只要求它把空间能力讲清楚：地理检索返回候选地点，POI provider 返回点位集合，空间分析工具返回 GeoJSON 或 artifact，图层 catalog 提供可用数据源。它还要说明边界条件，比如投影不一致时怎么处理，外部服务不可用时怎么报错，空结果时是不是允许文本交付。

测试子智能体也不是最后补洞，而是提前做工程真相校验。主智能体会让它围绕接口契约和关键流程写测试策略，比如地图搜索、图层切换、空间分析提交、任务状态恢复、澄清继续执行、错误提示是否可理解。尤其当前后端并行推进时，测试子智能体的反馈会像锚一样把系统拉回一致：字段有没有缺省值，空状态有没有处理，慢请求时 UI 会不会假死，用户选择澄清以后是不是还在同一个任务里。

部署子智能体则更像收口的人。主智能体会在中段就把它拉进来，而不是最后一天才部署。因为从零构建工作台，常见问题不是代码不会跑，而是本地能跑、服务器不行。所以部署子智能体要提前检查构建命令、环境变量、静态资源路径、数据库初始化、GIS 依赖和健康检查，把这些风险在开发阶段就暴露出来。

文档子智能体看似最轻，其实是保持团队一致性的关键。它不是最后美化 README，而是在每次接口、状态机或部署策略变动后，把系统怎么分工、谁是单一事实来源、失败怎么恢复、哪层不允许越权写清楚。这样后面不管是人接手，还是其他子智能体继续扩展功能，都不需要重新猜规则。

主智能体真正重要的地方，是管理结果回流。它会把子智能体的输出统一收成三类：可合并的代码、待决策的冲突点、需要继续拆分的未知项。比如后端说某个分析接口应该同步返回结果，前端说这样会卡住页面，GIS 说分析本身可能要跑二十秒。主智能体不会简单选边，而是把冲突上升成系统设计问题：是不是要改成异步 run，前端是不是订阅 event，测试要不要补超时和中断恢复，部署要不要支持后台任务队列。

最后，主智能体要做集成 review。它不会只看每个子智能体有没有完成自己的部分，而是专门检查跨边界链路：前端触发事件后，后端状态是否变化；GIS 分析结果是否能正确落到 artifact；测试是否覆盖失败路径；部署是否能把这些约定原样带到环境里。只有这些连接点都成立，项目才不是零散完成，而是系统可用。

这就是我想强调的多智能体协作。不是“多叫几个 AI 一起写”，而是让主智能体像一个工程导演，知道每个子智能体该站在哪，也知道最后镜头怎么接起来。

## 技术实现：把分工变成系统

分工不是停留在纸面上，接下来要把它变成可运行的系统。

这个系统的技术路线可以先用一句话概括：不是做一个能看地图的页面，而是做一个能理解空间问题、能调用工具、能留下过程证据的工作台。真正的 GIS 分析，不只是画点、画线、出结果，更重要的是让用户能把自然语言、空间检索、图层分析、人工确认和可追溯执行放进同一条工作流里。

前端选择 React，不是为了跟风，而是因为它适合把复杂交互状态拆清楚。地图工作台里最难的不是页面布局，而是状态来源的边界。哪些是服务端下发的 run、thread、event、artifact，哪些是用户正在编辑的查询条件，哪些是地图视图自己的临时态，哪些又是可以复用的图层配置，这些都要分开管理。React 让我们可以把 transcript、运行态、图层态和操作态分开，而不是揉成一个大对象。

地图渲染层用 MapLibre。它能把矢量底图、动态图层和高性能交互放在一起，适合做结果点位、面要素高亮、热区展示、图层切换和空间框选。用户在地图上每点一次、框一次、拖一次，背后都应该是明确的事件流，而不是页面自己猜用户想干什么。

后端用 FastAPI，核心价值是把 AI 调度、空间服务和业务 API 统一在一个清晰的服务边界里。FastAPI 适合把异步请求、流式事件和结构化响应组织起来，也方便我们把地图查询、POI 检索、GeoJSON 转换、分析任务触发这些能力拆成稳定接口。这里的关键不是接口多，而是接口都围绕同一个运行模型。

这个运行模型就是 run、thread、event、artifact。thread 是一次对话上下文，代表用户围绕某个任务的连续交流；run 是其中一次具体执行，可能是一次查询、一次地理分析、一次多步代理推理；event 记录运行过程，比如开始检索、完成 POI 命中、进入澄清、生成中间 GeoJSON；artifact 保存最终可见的结果，比如地图图层、分析报告、导出的 GeoJSON 或表格结果。

有了这套模型，用户不是只看到一句“分析完成”，而是能看到从澄清、检索、推理、生成图层到输出报告的完整链路。出了问题也能回放、定位、复现。对 AI 工作台来说，这不是工程细节，而是用户信任系统的基础。

AI 编排这一层，我会用 Deep Agents 和 LangChain tools 的组合思路来做。地图分析不是单轮问答，而是多工具、多步骤、多约束的协作过程。Deep Agents 适合承担“怎么推进任务”的高层策略，比如先澄清目标，再决定用 geosearch 还是 POI provider，再选择是否生成缓冲区或聚合图层；LangChain tools 则适合把每个具体动作工具化，比如空间查询、地址解析、GeoJSON 生成、图层筛选、统计聚合和报告摘要。

这样 AI 就不只是会说，而是会做。用户输入“查询东京羽田机场 3 公里范围内的医院”，系统先用 geosearch 解析地点，再用 POI provider 查询周边医院，再把结果转成 GeoJSON，保存成 artifact，最后让 MapLibre 显示结果。每一步都能落到 event 上，用户知道系统为什么这么做，模型也能在下一步继续利用这些中间结果，而不是每次重头开始。

GeoJSON 是前后端之间最重要的空间数据中枢。无论是点、线、面，还是筛选出的 POI、聚合后的区域、路径分析结果，都尽量转成 GeoJSON 或类似结构，这样前端和 MapLibre 能直接消费，后端也能明确校验和转换。GeoJSON 的好处是足够通用，又贴近地图展示，让分析结果天然具备可视化属性。

图层 catalog 也是系统里很重要的一层。很多地图产品的问题，不是没有图层，而是图层没有秩序。catalog 的职责，是把可用图层变成可理解、可组合、可复用的资产。它要记录图层类型、数据源、样式、权限、适用场景、是否可编辑、是否可被 AI 自动调用。这样用户切换图层时，不只是打开一个开关，而是在选择一组分析语义。AI 也可以基于 catalog 判断哪些图层适合叠加，哪些图层不该混用，哪些图层只能只读。

澄清机制则是这个工作台能否真正可用的分水岭。空间任务天然有歧义。比如用户说“帮我找附近人流多的地方”，这里的“附近”到底是多少米，“人流多”是哪个时间段，“地方”指商圈还是 POI 群，“分析范围”是城市中心还是当前视野？如果没有澄清机制，AI 很容易直接给出一个看似合理、其实毫无业务约束的答案。所以系统不是一上来就回答，而是在 run 的早期通过 event 触发澄清。目标不清、范围不清、时间窗不清、数据源不清时，系统先回问，等用户确认后再继续。

如果分阶段实现，第一阶段先做最小可用闭环。用户能从一句话开始，经过澄清、检索、生成图层、在地图上展示结果，再拿到一个 artifact。前端先把页面、地图容器、会话面板和图层面板搭起来，后端先把 run/thread/event/artifact 模型跑通，AI 层先接入一个能调用 geosearch 和 POI provider 的最小工具链。只要这个闭环成立，系统就从概念变成可操作。

第二阶段，把能力从单次查询推进到多步骤分析。Deep Agents 开始自动拆解复杂请求，先从 geosearch 确认范围，再从 POI provider 拉取候选点位，再根据规则做空间过滤、聚合统计和结果解释。前端支持事件流式展示，让用户看到模型正在做什么，而不是等一个静态答案。

第三阶段，做可复用和可扩展。把常见分析模式固化成模板，把图层、工具和提示词策略模块化，把 POI provider、地理编码服务、外部 API 都放进统一适配层。这样新的数据源接进来，不需要重写整个系统，只要实现标准工具接口和 GeoJSON 输出即可。

所以这条技术路线的核心不是某一个框架，而是把 React、FastAPI、MapLibre、Deep Agents、LangChain tools、GeoJSON、geosearch、POI provider、图层 catalog 和澄清机制，组织成一个围绕 run、thread、event、artifact 的统一系统。

## 让 Codex 做代码 review 和体验 review

代码写完只是开始。真正决定这个系统能不能演示、能不能交付的，是 review 时它能不能面对真实输入和真实错误。

我会让 Codex 先做代码 review，而且不是只看语法问题，而是专门盯几个原则：图层是不是写死，业务逻辑有没有临时 hack，fallback 是真正容错还是掩盖主路径问题，数据流是不是单一事实来源，前端、后端和数据库有没有各说各话。

比如图层这件事，很多项目一开始都会写死图层列表，前端直接拿一组固定配置去渲染，短期看省事，长期会变成维护灾难。这次的要求很明确：图层不能写死，必须从数据和配置驱动。Codex 在 review 时要盯住这个边界，提醒哪些图层是运行时决定的，哪些来自后端能力发现，哪些只是调试态。

fallback 也要重点 review。我们希望保留的 fallback 很少，而且每一个都要有理由。如果一个分析任务的图层元数据缺失，可以允许显示空态和提示，让用户继续补充条件，但不能偷偷塞一个看起来差不多的默认图层，把真实状态伪装成已完成。如果模型没有给出可执行计划，可以进入 plan repair，但不能把另一套规则执行结果伪装成模型成功。

体验 review 的核心，不是页面漂不漂亮，而是用户能不能真正完成一次自然语言驱动的 GIS 分析。用户不应该先学习复杂表单，也不应该理解系统内部有哪些固定操作。用户直接说“帮我找出某区域周边新增的高密度热区，并和上个月做对比”，系统应该先理解意图，再判断条件是否足够，再决定要不要追问。

Codex 在体验 review 里要检查交互链路有没有断。是不是用户输入之后，系统先给出不可靠结论；是不是本来应该澄清的地方，被前端默认值悄悄吞掉了；是不是结果页只展示了一个漂亮图，但没有把查询依据、图层来源、筛选条件和时间范围讲清楚。

主聊天区也要 review。不能把每一次 tool.started、tool.completed、loop.updated 都做成一张大卡片。那样用户看到的不是智能助手，而是一堆技术日志。更自然的方式，是像 Codex 一样，主线用中文说明系统正在做什么，工具记录合并成“已运行几个命令”的折叠块，用户需要时再展开。

通过这样的 review，Codex 不只是写代码，而是在帮我不断把系统从“能跑”推向“合理”。

## 测试联调

测试联调不是只看单元测试绿不绿，而是把系统开到复杂路况里，看它会不会在转弯、刹车、上坡、切换模式的时候出问题。

我会让 Codex 一边改代码，一边推动我们把关键路径补成可验证的场景。自然语言查询要走到后端解析、图层检索、空间分析、结果回传、前端呈现，这中间任何一环都不能靠想当然。测试不只是检查“返回成功”，还要确认条件不足时会不会触发澄清，图层元数据缺失时是否进入明确空态，分析超时或结果不完整时有没有合适的恢复行为。

真实联调时，我最关心三件事。

第一，主流程能不能跑通。比如输入“查询东京羽田机场 3 公里范围内的医院”，系统能不能解析地点，能不能查到医院，能不能生成 GeoJSON，地图能不能显示结果，右侧能不能给出摘要。

第二，失败时是不是说人话。如果地点找不到，就说地点没有找到；如果有多个候选，就让用户选择；如果外部 POI 服务不可用，就说明检索服务暂时不可用。不能只把 Python 异常或 JSON 原样扔给用户。

第三，上下文是不是稳定。用户选择澄清时，应该继续当前任务，而不是新开一个任务；用户输入相同文本时，如果不是从历史任务继续，就应该创建新的会话；任务历史要能保存标题、用户问题、运行状态和结果摘要。

这类产品最怕前端和后端各自以为对方会兜底，最后整个链路没有一个真正的单一事实来源。所以联调时要刻意把边界弄清楚：运行时状态归谁管，查询条件归谁确认，图层元数据从哪来，哪些信息可以本地编辑，哪些必须回到服务端重新确认。

## 现场演示

现场演示时，我不会去炫技堆场景，而是让大家看到一条完整、真实、可理解的路径。

开场先讲方法：这个项目不是直接让 Codex 写代码，而是先让它生成计划，再让它 review 计划，然后由主智能体指挥多个子智能体分工实现，最后统一 review、联调和交付。这样观众能先理解，这不是一次碰运气生成，而是一个可以管理的工程过程。

接着展示核心架构。前端是 React 工作台，后端是 FastAPI，地图用 MapLibre，Agent 用 Deep Agents 和 LangChain tools，空间结果用 GeoJSON，运行状态围绕 run、thread、event、artifact。这里不需要罗列太多技术名词，重点是让大家明白每层解决什么问题。

然后进入核心查询。我会输入一条接近真实用户会说的话，比如“查询东京羽田机场 3 公里范围内的医院”。系统先理解意图，再解析地点，再调用 POI provider 查询医院，再生成 GeoJSON，最后把结果显示到地图上。用户可以展开命令记录，看到实际调用过哪些工具。

再演示澄清场景。比如输入一个有歧义的地点，系统不应该随便执行，而是给出候选，让用户选择。用户选择以后，任务继续推进。这一段能体现系统不是写死的，而是真的把地点解析和澄清作为主流程。

最后展示数据源和扩展能力。说明图层不是写死在代码里的，而是来自 catalog；外部 POI provider、地理编码服务、PostGIS Processing、地图服务发布都可以按工具和数据源继续接入。现在演示版本虽然小，但架构是朝真实系统设计的。

现场演示最重要的观感，不是答案很快，而是系统像一个懂 GIS 的协作者。它不会乱猜，不会假装知道，也不会拿一个写死图层凑数。它会根据当前任务动态选数据源，必要时反问，结果出来之后还能继续追问。

## 交付收尾

最后是交付。

一个项目从能演示到能交付，中间差的往往不是代码量，而是部署链路是不是稳。所以我会让 Codex 把部署脚本、启动步骤、环境变量、构建流程都整理清楚，不再依赖“我本机上试过没问题”这种不可交付的状态。

部署子智能体会检查脚本是否遗漏依赖，是否存在路径写死，是否有临时环境配置没清理干净，是否把测试数据和生产配置混在一起。目标很朴素，但非常关键：让别人拿到项目后，能按脚本跑起来，能按说明演示出来，出现问题时能定位到具体边界，而不是在一堆 hack 里猜答案。

所以这次用 Codex 从 0 构建 AI GIS 地图分析工作台，核心不只是“做出了一个地图 AI 产品”，而是把一个工程过程真正跑通了。先用 Codex 生成计划，把模糊需求变成工程语言；再用 Codex review，把写死图层、乱 fallback、状态不清这些风险提前挑出来；再用主智能体指挥子智能体分工，把复杂系统拆成可管理的工作面；然后用 React、FastAPI、MapLibre、Deep Agents、GeoJSON 和工具层完成核心闭环；最后通过 review、测试、联调和部署，把系统从能跑推进到能交付。

对我来说，这才是 Codex 最值得展示的地方。它不是替我写几行代码，而是帮助我把一个模糊想法，经过计划、评审、分工、实现、测试和修正，推进成一个可以运行、可以解释、可以继续扩展的工程项目。
